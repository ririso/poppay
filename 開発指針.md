# PayPay延長料金支払いWebページ 技術選定・実装方針

> 作成日：2026-05-31
> ステータス：設計ドラフト
> 想定開発者：バックエンド（Python / Java）が主軸、フロントは学習中

---

## 1. 何を作るか

カウンセリングなどの「延長料金」をその場でPayPayで支払ってもらうためのWebページ。

```
① 管理者が金額を入力
        ↓
② PayPayのQRコードが生成される
        ↓
③ 支払いユーザーがPayPayアプリでQRを読み取る
        ↓
④ ユーザーが支払いを実行
        ↓
⑤ 管理者画面に「支払い完了」が反映される
```

最終的には、この仕組みをパッケージ化して複数の事業者（占い師など）が各自で使えるSaaSへ発展させる構想がある。本ドキュメントはまず「単一事業者で動く最小構成」に絞り、拡張ポイントを併記する。

---

## 2. PayPayの決済方式の選択

PayPay Open Payment API（OPA）には複数の決済方式がある。今回の「QRを出してユーザーが読む」用途には **動的ユーザースキャン（Dynamic QR Code）** が最適。

| 方式                 | 概要                                                       | 今回の適合度                         |
| -------------------- | ---------------------------------------------------------- | ------------------------------------ |
| 動的ユーザースキャン | 動的なQRコードを生成し、ユーザーがアプリで読み取って決済   | ◎ 最適                               |
| アプリコール         | ユーザー端末でPayPayアプリを呼び出して決済（同一端末向け） | △ 対面で別端末が読むなら不向き       |
| ネイティブペイメント | 事前認可でウォレットから直接引き落とし                     | × 事前のアカウントリンクが必要で過剰 |
| 継続課金             | サブスク向けの定期決済                                     | × 用途が違う                         |

動的ユーザースキャンは、QRコード生成APIを叩くと「PayPayキャッシャーページのURL」と「ディープリンク」が返ってくる。これをQRコード画像にして画面表示すればよい。

## 3. 技術スタック

バックエンドが主軸という前提で、フロントの学習コストを抑えつつ一人で完結できる構成を選ぶ。

| 層               | 採用候補                                                  | 理由                                                                 |
| ---------------- | --------------------------------------------------------- | -------------------------------------------------------------------- |
| フロント＋API    | Next.js（App Router）                                     | 画面とサーバー処理を1プロジェクトで完結。APIキーをサーバー側に隠せる |
| 言語             | TypeScript                                                | PayPayのNode.js SDKがそのまま使える                                  |
| 決済             | PayPay OPA（Node.js SDK `@paypayopa/paypayopa-sdk-node`） | 公式SDK。署名処理を自前実装せずに済む                                |
| QR描画           | `qrcode`（npm）                                           | APIが返すURLをQR画像化                                               |
| DB               | Supabase（PostgreSQL）                                    | 取引履歴の保存。無料枠で十分。後のSaaS化でも流用可                   |
| リアルタイム反映 | Supabase Realtime or ポーリング                           | 「支払い完了」を管理画面へ反映                                       |
| ホスティング     | Vercel                                                    | Next.jsと相性が良く、Webhook受け口もサーバーレスで持てる             |

### なぜPythonではなくNode.js（TypeScript）か

PayPayはPython/Node.js/PHP/Javaの4言語でSDKを提供しているため、Pythonでも実装は可能。ただし今回は「フロントの学習」も兼ねたい意向があるため、フロントとバックエンドを1つの言語（TypeScript）で書けるNext.js構成を推す。Python（FastAPI）でバックエンドを分離する構成も問題なく成立するので、慣れた方を選んでもよい。

## 4. 決済フロー（技術レベル）

```
[管理者画面]                [Next.js APIルート]            [PayPay OPA]
   金額入力
   QR生成ボタン  ──POST──▶  createQRCode()  ──API──▶  QRコード発行
                                                          (url / deeplink / codeId)
   QR画像を表示  ◀────────  url を返す  ◀──────────────

[ユーザー端末]
   PayPayアプリでQRスキャン ──▶ PayPay上で支払い実行

[支払い完了の検知]
   方式A: Webhook で COMPLETED 通知を受信（推奨）
   方式B: getPaymentDetails を2〜3秒間隔でポーリング

   → DBの取引ステータスを更新 → 管理画面へ反映
```

### 4-1. QRコード生成（サーバー側）

`merchantPaymentId`（加盟店側で発行する一意の取引ID）を必ず毎回ユニークに採番する。使い回すと決済が衝突する。`codeType` は `ORDER_QR`、`amount` に延長料金、`orderDescription` にメモ（例：30分延長）をセットする。レスポンスの `data.url` をQR化して表示する。

### 4-2. 支払い完了の検知

PayPay公式は「リダイレクトが返らないケース（ユーザーがアプリを閉じた等）があるため、`getPaymentDetails` での確認」を案内している。確実性のため、以下の二段構えが堅実。

- 主：Webhook（トランザクションのステータスが `COMPLETED` に変わった際に通知が届く）を受信し、200 OKを返す
- 補：管理画面側で `getPaymentDetails` を2〜3秒間隔でポーリングし、Webhookの取りこぼしを補完
  Webhookはセキュリティ上、PayPayのIPアドレスのホワイトリスト登録が強く推奨されている。また2秒間応答がないとPayPay側でタイムアウトし再送されるため、Webhookハンドラは「受信して即200を返し、重い処理は非同期で」が原則。

## 5. 環境構成（Sandbox → 本番）

PayPayは3つの環境を持つ。審査前でもSandboxは使えるので、加盟店審査を待つ間に開発・テストを並行できる。

| 環境         | エンドポイント                          | 用途                                 |
| ------------ | --------------------------------------- | ------------------------------------ |
| Sandbox      | `https://stg-api.sandbox.paypay.ne.jp/` | 開発・テスト（実際のお金は動かない） |
| ステージング | `https://stg-api.paypay.ne.jp/`         | 本番前検証                           |
| 本番         | `https://api.paypay.ne.jp/`             | 実決済                               |

SDKでは `productionMode: false/true` で切り替える。APIキー・シークレット・マーチャントIDは環境ごとに異なる。

---

## 6. セキュリティ方針

APIキー（API Key / API Secret）は加盟店としての操作権限を持つ。漏洩すると返金APIの悪用や売上データ閲覧などのリスクがあるため、扱いを厳格にする。

- APIキーは絶対にフロントエンドのコードに書かない。Next.jsでは `NEXT_PUBLIC_` プレフィックスを付けるとブラウザに露出するので、PayPayのキーには付けない
- ソースコードに直接書かず、環境変数（Vercelの環境変数 / Supabaseのシークレット管理）で管理する
- `.env` ファイルは `.gitignore` に入れ、GitHubにpushしない
- QRコード生成・決済確認・返金などキーを使う処理はすべてサーバーサイド（APIルート）に閉じる
- 定期的なキーのローテーションを行う
- Webhook受信時はPayPayのIPホワイトリストを設定する
- 管理者画面には認証（最低限のパスワード、できればSupabase Auth）をかける

## 7. データ設計（最小構成）

取引を1テーブルで管理する。SaaS化を見据えるなら最初から `tenant_id`（事業者ID）を持たせておくと後が楽。

| カラム              | 型          | 説明                                     |
| ------------------- | ----------- | ---------------------------------------- |
| id                  | uuid        | 主キー                                   |
| tenant_id           | uuid        | 事業者ID（SaaS化用。当面は固定値でも可） |
| merchant_payment_id | text        | PayPayへ渡す一意の取引ID                 |
| amount              | integer     | 金額（円）                               |
| description         | text        | メモ（例：30分延長）                     |
| status              | text        | CREATED / COMPLETED / FAILED / EXPIRED   |
| paypay_code_id      | text        | PayPayが返すcodeId                       |
| created_at          | timestamptz | 生成日時                                 |
| paid_at             | timestamptz | 支払い完了日時                           |

## 8. 開発フェーズ

### Phase 0 — 準備

- [ ] PayPay for Developers アカウント作成
- [ ] 加盟店審査の申請（審査に1週間程度。早めに着手）
- [ ] Sandbox環境のAPIキー取得

### Phase 1 — Sandboxで動く最小版

- [ ] Next.jsプロジェクト作成
- [ ] 管理者画面（金額入力 → QR生成ボタン）
- [ ] APIルートでQRコード生成（SDK利用）
- [ ] QRコードを画面表示
- [ ] `getPaymentDetails` ポーリングで支払い完了を検知
- [ ] 取引をSupabaseに保存

### Phase 2 — 堅牢化

- [ ] Webhook受信エンドポイントを実装（主の検知手段に）
- [ ] PayPay IPホワイトリスト設定
- [ ] 管理者画面に認証を追加
- [ ] 支払い履歴一覧
- [ ] 本番環境APIキーへの切り替え・本番テスト

### Phase 3 — SaaS化（将来構想）

- [ ] マルチテナント化（事業者ごとのアカウント・店舗）
- [ ] 各事業者が自分のPayPay加盟店情報を登録できる仕組み
- [ ] クレカ決済（Stripe）の追加
- [ ] 出品・在庫・売上管理
  > 注：SaaS化で各事業者が個別に売上を受け取る形にする場合、PayPayにはStripe Connectのような「プラットフォームが一括管理して分配する」仕組みがない。各事業者がそれぞれPayPay加盟店申請を行う前提になる点に注意。
